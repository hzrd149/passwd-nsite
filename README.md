# passwd-nsite

`passwd-nsite` is a small Vite + React app for publishing a password-protected nsite.

It has two jobs:

1. Serve a locked shell that downloads `/site.7z`, asks for a password, and unlocks the site in the browser.
2. Publish a locked nsite by building `site.7z`, uploading all required blobs to Blossom, and publishing a named-site manifest event.

## User Overview

This app has two hash routes:

- `#/` is the lock and unlock flow.
- `#/publish` is the publish flow.

The normal user flow is:

1. Visit the site.
2. The app checks whether the unlocked site is already loaded in IndexedDB.
3. If not, it fetches `/site.7z`.
4. The user enters the password.
5. The browser extracts the archive, stores the files locally, enables the router service worker, and opens the unlocked site.

Locking the site again removes the service worker and deletes the stored files.

## How The Locked Site Works

The lock and unlock flow is implemented in `src/pages/HomePage.tsx`.

- Locked content lives in `/site.7z`.
- The archive is decrypted and extracted in the browser with `7z-wasm` from `src/lib/7zip.ts`.
- Extracted files are written into IndexedDB by `src/router/storage.ts`.
- The service worker in `src/router/index.ts` serves those stored files when router mode is `"on"`.

Important behavior:

- Stored file paths are normalized to absolute paths like `/index.html`.
- Requests for `/`, `/about`, and `/about/` fall back to `index.html` when appropriate.
- If a requested file is missing, `/404.html` is used if present.
- Locking the site is destructive: it unregisters the service worker and deletes the router database.

## How Publishing Works

The publish flow is implemented in `src/pages/PublishPage.tsx` and the helpers in:

- `src/lib/publish.ts`
- `src/lib/blossom.ts`
- `src/lib/nostr.ts`

When publishing from the web UI, the app:

1. Connects to a NIP-07 signer.
2. Reads a selected site folder.
3. Builds an encrypted `site.7z`.
4. Fetches `/assets.json` from the current build.
5. Fetches every file listed in `/assets.json` and verifies its SHA-256 hash.
6. Adds `/assets.json` itself.
7. Adds `/site.7z`.
8. Uploads all of those blobs to one or more Blossom servers.
9. Signs a named-site manifest event of kind `35128`.
10. Publishes that manifest to the user's outbox relays.

The published nsite is a named site, not a root site.

## What Must Be Published

The final published bundle is not only `site.7z`.

It includes:

- Every file in the current production build listed by `/assets.json`
- `/assets.json`
- `/site.7z`

In a production deployment, `vite.config.ts` intentionally removes `public/site.7z` from `dist`, so `site.7z` must be created separately for each locked publish.

## Build Output

Build the app with:

```bash
pnpm build
```

That produces:

- `dist/index.html`
- `dist/router.js`
- `dist/assets/*`
- `dist/assets.json`

`dist/assets.json` is generated by Vite and contains a flat list of:

```json
[
  {
    "path": "/index.html",
    "sha256": "..."
  }
]
```

The publish flow trusts that file as the authoritative list of build assets that belong to the locked shell.

## Data Model For Agents

### Archive Paths

Files inside `site.7z` use relative paths like:

- `index.html`
- `blog/index.html`
- `assets/logo.png`

If all input files share one top-level folder, that shared root is stripped before archiving and before manifest generation.

Example:

- `my-site/index.html` becomes `index.html`
- `my-site/blog/post.html` becomes `blog/post.html`

### Manifest Paths

Manifest `path` tags always use absolute paths like:

- `/index.html`
- `/blog/index.html`
- `/assets/logo.png`

### Site ID Rules

Named-site ids are validated in `src/lib/publish.ts`.

They must:

- match `^[a-z0-9-]{1,13}$`
- be lowercase
- not end with `-`

### Nsite Event Shape

This app publishes kind `35128` events with:

- one `d` tag for the site id
- one or more `path` tags
- one aggregate `x` tag
- zero or more `server` tags
- optional `title`
- optional `description`

Example:

```json
{
  "kind": 35128,
  "content": "",
  "tags": [
    ["d", "passwd"],
    ["path", "/index.html", "<sha256>"],
    ["path", "/assets/index.js", "<sha256>"],
    ["path", "/site.7z", "<sha256>"],
    ["x", "<aggregate-sha256>", "aggregate"],
    ["server", "https://cdn.example.com"],
    ["title", "Password Protected nsites"],
    ["description", "Deploy an nsite with a password"]
  ]
}
```

## Aggregate Hash

The aggregate hash identifies the exact site version.

It is computed from the `path` tags only:

1. For each path tag, produce a line in the form `<sha256> <absolute-path>\n`.
2. Sort those lines lexicographically.
3. Concatenate them.
4. SHA-256 hash the result.
5. Encode the digest as lowercase hex.

Example:

```text
186ea5fd... /index.html
fedcba09... /site.7z
```

Example shell pipeline:

```bash
sort paths.txt | sha256sum
```

Where `paths.txt` contains one line per published path in the exact format above.

## Blossom Upload Model

Uploads are handled in `src/lib/blossom.ts`.

The app uploads every blob to every target server.

A server only counts as usable if every required blob is present there.

The browser flow uses signed Blossom upload auth events:

- auth kind: `24242`
- tags include:
  - `['t', 'upload']`
  - one `['x', '<sha256>']` per blob hash in the batch
  - `['expiration', '<unix-seconds>']`
  - `['client', 'passwd-nsite']`

Each blob upload is:

1. `HEAD <server>/<sha256>` to check whether it already exists
2. `PUT <server>/upload` with `Authorization: Nostr <base64-json-event>` if needed

## CLI Publish Workflow

This section is for users and agents that want to reproduce the web publish flow from the command line.

The exact signer and Blossom tooling may vary, but the required steps do not.

### 1. Build The Locked Shell

```bash
pnpm build
```

You now have the files listed in `dist/assets.json`.

### 2. Build `site.7z`

Create an encrypted archive of the site you want to protect.

Use the 7z CLI with password protection and header encryption:

```bash
7z a -t7z site.7z ./my-site/* -pYOUR_PASSWORD -mhe=on
```

Notes:

- `-mhe=on` matters. The app does header-encrypted archives in `src/lib/7zip.ts`.
- The archive should contain the site files at the root, not an unwanted wrapper directory.
- If your command archives a parent folder, the app may still work because it strips one shared root folder on unlock, but it is better to keep the archive layout clean.

To inspect the archive:

```bash
7z l -slt site.7z
```

### 3. Hash The Published Files

You need hashes for:

- every file listed in `dist/assets.json`
- `dist/assets.json` itself
- `site.7z`

Example:

```bash
sha256sum dist/index.html
sha256sum dist/router.js
sha256sum dist/assets/*
sha256sum dist/assets.json
sha256sum site.7z
```

Your manifest must include absolute paths like `/index.html`, `/router.js`, `/assets/...`, `/assets.json`, and `/site.7z`.

### 4. Upload All Blobs To Blossom

For each target Blossom server, upload:

- each build asset from `dist/assets.json`
- `dist/assets.json`
- `site.7z`

The web app first checks whether a blob already exists with:

```bash
curl -I https://your-blossom-server.example/<sha256>
```

If the blob does not exist, it uploads it with a signed auth header to:

```text
PUT https://your-blossom-server.example/upload
```

The exact auth signing command depends on your signer tool, but the signed event must authorize upload for the blob hashes you want to send.

Pseudo-flow:

```text
for each blossom server:
  for each blob:
    HEAD /<sha256>
    if missing:
      sign kind 24242 upload auth event
      PUT /upload with Authorization: Nostr <base64-json-event>
```

At least one Blossom server must end up holding the full set of blobs.

### 5. Build The Named-Site Manifest Event

Create a kind `35128` event with:

- `content` set to `""`
- `d` tag set to the site id
- one `path` tag per published path
- one aggregate `x` tag
- one `server` tag per Blossom server that has the full bundle
- optional `title` and `description`

The app sorts path tags by path before building the event.

Example tag set:

```json
[
  ["d", "passwd"],
  ["path", "/assets.json", "<sha256>"],
  ["path", "/assets/index-CTXYBDgP.js", "<sha256>"],
  ["path", "/index.html", "<sha256>"],
  ["path", "/router.js", "<sha256>"],
  ["path", "/site.7z", "<sha256>"],
  ["x", "<aggregate-sha256>", "aggregate"],
  ["server", "https://cdn.example.com"],
  ["title", "Password Protected nsites"],
  ["description", "Deploy an nsite with a password"]
]
```

### 6. Publish The Event To Relays

Publish the signed kind `35128` event to your outbox relays.

The browser version loads relay targets from:

- kind `10002` relay list metadata when available
- otherwise defaults in `src/const.ts`

If you are using `nak`, the final step is conceptually:

```bash
nak event --kind 35128 ... wss://relay.example.com wss://relay2.example.com
```

The exact flags depend on how you sign events in your CLI environment, but the payload must match the manifest shape above.

## Minimal Checklist For Agents

If you are automating a publish, make sure all of these are true:

1. Run `pnpm build` and read `dist/assets.json`.
2. Create an encrypted `site.7z` with `7z` and `-mhe=on`.
3. Hash every build file, `dist/assets.json`, and `site.7z`.
4. Upload all blobs to at least one Blossom server.
5. Build a kind `35128` event with `d`, `path`, `x`, and `server` tags.
6. Compute the aggregate hash from sorted `<sha256> <path>` lines.
7. Publish the signed event to at least one relay.
